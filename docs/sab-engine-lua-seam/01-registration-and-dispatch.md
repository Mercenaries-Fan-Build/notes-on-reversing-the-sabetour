# Registration and dispatch: how a C function becomes a callable Lua name

## What this establishes

The 898-name Lua surface is generated by a **C++ class template**, not a hand-written table.
Every binding is one instantiation of `LuaGlueFunctor0<&F>` (for `void F(lua_State*)`) or
`LuaGlueFunctor0R<int,&F>` (for `int F(lua_State*)`). MSVC's RTTI for those instantiations
leaves the C++ function symbol in the binary **in the clear**, and the Lua-visible name is a
plain string in a per-binding registration stanza.

Because of that, the whole seam is recoverable statically. This doc resolves the
`0x00716a00-0x00716c00` anomaly, gives the five-layer structure from Lua name to C
implementation, and ships a reproducible dumper that emits all 898 rows of
`{table, lua_name, cpp_symbol, impl_va}`:

- Tool: [`tools/dump_lua_registration.py`](../../tools/dump_lua_registration.py)
- Output: [`data/lua_registration_map.tsv`](../../data/lua_registration_map.tsv) (898 rows, 26 tables)

The single most consequential finding for the rest of this effort:
**[`data/lua_bindings.txt`](../../data/lua_bindings.txt) is a list of C++ symbols, not of
Lua-callable names.** 256 of the 898 differ. `ActorRagdoll` is not callable from Lua;
`Actor.Ragdoll` is. See [Naming](#the-name-lives-in-two-places-and-they-differ).

Sibling docs: the marshalling primitives (`FUN_006f8470` / `FUN_006f71a0` / `FUN_006f6ec0`)
that appear *inside* each `F` are [02-marshalling-abi.md](02-marshalling-abi.md); the handle
table is [03-handle-and-object-model.md](03-handle-and-object-model.md); when the registries
below are actually drained into live Lua tables is
[04-vm-lifecycle-and-script-objects.md](04-vm-lifecycle-and-script-objects.md); the
`WRAPPER_*.lua` layer above these names is
[06-lua-side-wrapper-layer.md](06-lua-side-wrapper-layer.md).

## The anomaly: 0x00716a00-0x00716c00 is a thunk array

Ghidra reports `FUN_00714230` (ActorRagdoll) with `callers=[0x00716b25]`, and 0x00716b25 lies
inside no defined function â€” the last `FUN_` in the page is `FUN_007166a0` (ends ~0x007166e3).
It is neither a registration table nor a Ghidra bug. **It is an array of 32-byte
`lua_CFunction` adapter thunks that nothing calls**, so Ghidra's auto-analysis never created
functions there: the only reference to each is a data pointer from a vtable.

Reading the bytes at file offset 0x315c00 (VA 0x00716a00) shows the stanza repeating verbatim
every 32 bytes:

```
8b 44 24 04      mov  eax, [esp+4]        ; lua_State* L
50               push eax
e8 <rel32>       call <impl F>
83 c4 04         add  esp, 4
b8 01 00 00 00   mov  eax, 1
c3               ret
cc x14           int3 padding to 32-byte alignment
```

18 bytes of code, padded to 32. That resolves all three probes exactly â€” the reported caller VA
is simply the address of the `call` at `thunk+5`:

| Ghidra says | thunk base | binding |
|---|---|---|
| `FUN_00714320  callers=[0x00716a85]` | 0x00716a80 | `Actor.SetRunsFromFire` |
| `FUN_00714230  callers=[0x00716b25]` | 0x00716b20 | `Actor.Ragdoll` |
| `FUN_00714170  callers=[0x00716b45]` | 0x00716b40 | `Actor.SetDistantRagdollSound` |

The gap 0x007166e3 â†’ 0x00717760 contains these thunks *and* the getter stubs and constructors
described below. The same pattern recurs across the binary â€” this is not special to
0x716xxx.

**Confidence: confirmed** (byte-level; the three caller VAs are arithmetically identical to
`thunk+5`, and `Actor.Ragdoll` independently matches the `"ActorRagdoll"` assertion string in
`FUN_00714230`).

## The five layers

Per binding, MSVC emits five artifacts. Worked example, `Actor.Ragdoll`:

| # | Layer | VA | Content |
|---|---|---|---|
| 1 | RTTI type descriptor | 0x01125f08 | `.?AV?$LuaGlueFunctor0@$1?ActorRagdoll@@YAXPAUlua_State@@@Z@@` |
| 2 | RTTI complete-object-locator | 0x010bb848 | `+0x0c` â†’ the type descriptor |
| 3 | vtable (1 slot), COL at `[vt-4]` | 0x00fde930 | slot 0 â†’ the getter stub |
| 4 | getter stub | 0x00717390 | `mov eax, 0x716b20; ret` |
| 5 | adapter thunk | 0x00716b20 | `call 0x714230; mov eax,1; ret` |
| â€” | implementation `F` | 0x00714230 | `ActorRagdoll(lua_State*)` |

So `LuaGlueFunctor0` is an abstract base (`.?AVLuaGlueFunctor@@`, 1 non-template descriptor)
with a single virtual method that returns the `lua_CFunction` for its template argument. The
functor object itself is **4 bytes â€” just the vptr**; the name is a `static` member (below).

Counts: 899 `LuaGlue*` type descriptors = 722 `LuaGlueFunctor0` + 176 `LuaGlueFunctor0R` + 1
non-template base. 722 + 176 = **898**, matching `lua_bindings.txt` exactly.

`LuaGlueFunctor0R` carries the return type as an extra template argument
(`LuaGlueFunctor0R@H$1?...`, `H` = `int`). Because `int F(lua_State*)` **already has the
`lua_CFunction` signature**, no adapter is needed and layer 5 degenerates to a plain
`jmp <impl>`.

Three thunk shapes observed across the 898:

| shape | count | code | meaning |
|---|---|---|---|
| `adapter` | 640 | `push L; call F; add esp,4; mov eax,1; ret` | `void F` + separate impl |
| `jmp` | 172 | `jmp F` | `int F`, returns its own nresults |
| `inlined` | 86 | body, then `mov eax,1; ret` | `F` inlined; the adapter **is** the impl |

The 86 `inlined` cases have no distinct implementation function â€” e.g. the thunk for
`Actor.PlayerBlockSabLighting` at 0x00716330 is the whole binding:
`mov eax,[0x1240328]; or byte [eax+0x16e8], 0x80; mov eax,1; ret`. For these the tool reports
`impl_va == thunk_va`, which is honest rather than a failure.

**Confidence: confirmed** for the layer structure and counts (RTTI strings + byte patterns).

## Registration is imperative and per-table

**There is no static `{name, fnptr}` array anywhere in `.rdata`/`.data`.** I checked directly:
scanning every section for aligned DWORDs equal to a thunk VA yields only two hits, and both
are coincidental `call rel32` encodings inside `.text`. Scanning for `push imm32` of a thunk VA
yields zero. The only reference to each thunk is the `mov eax, imm32` getter stub, and the only
reference to each vtable is a `mov dword ptr [reg], <vtable>` constructor store.

Registration is instead **an 84-byte stanza per binding**, emitted inline into a static
initializer. Disassembled at 0x0070d6eb (`Actor.Ragdoll`):

```
6a 01 / 6a 14            push 1; push 0x14
e8 ...                   call 0xdb39e0            ; operator new(0x14) -> list node (edi)
89 70 08 / 0c / 10       zero node+8/+0xc/+0x10
6a 01 / 6a 04            push 1; push 4
c7 47 04 <str>           mov [edi+4], "Ragdoll"   ; node.name
e8 ...                   call 0xdb39e0            ; operator new(4) -> the functor
c7 00 30 e9 fd 00        mov [eax], 0x00fde930    ; vptr = LuaGlueFunctor0<&ActorRagdoll> vtable
c7 05 1c d8 42 01 <str>  mov [0x0142d81c], "Ragdoll"   ; the template's static s_name
57 / 8b cb / 89 07       push edi; mov ecx, ebx; mov [edi], eax  ; node.fn = functor
e8 ...                   call 0x006f6660          ; registry->insert(node)
```

`FUN_006f6660` is a plain intrusive doubly-linked-list insert that also bumps a count at
`head+0x10` â€” the registry is a **list, not a hash table**, at least at build time.

The table identity is `ebx`, and it is established once at the top of each initializer. At
0x0070caf0:

```
push ebx / push esi
6a 01 / 6a 3c            push 1; push 0x3c
e8 ...                   call 0xdb39e0            ; operator new(0x3c) -> a fresh registry
68 f0 d7 fd 00           push 0x00fdd7f0          ; "Actor"      <-- the table name
8b c8                    mov ecx, eax
e8 10 9b fe ff           call 0x006f6620          ; registry ctor(name)
8b d8                    mov ebx, eax
```

So each initializer news a 0x3c-byte registry, names it, and links its own bindings into it.
The 898 stanzas group into **26 contiguous runs** â€” one per Script translation unit â€” and each
run yields exactly one table name. They sum to 898:

| table | n | table | n | table | n |
|---|---|---|---|---|---|
| Util | 173 | Train | 31 | Squad | 18 |
| Actor | 111 | AttractionPt | 25 | SaveLoad | 13 |
| Vehicle | 80 | Checkpoint | 24 | Sabotage | 8 |
| Combat | 65 | Nav | 23 | Sensory | 8 |
| HUD | 57 | Trigger | 23 | Filter | 3 |
| Object | 44 | Freeplay | 21 | Damage | 2 |
| Render | 42 | Sound | 21 | Searchlight | 2 |
| Suspicion | 34 | FocusPt | 18 | Zone | 2 |
| Cin | 32 | Inventory | 18 | | |

**It is one namespace per table, not one global namespace.** The decisive evidence is name
collision: 16 Lua names are registered more than once, and they only make sense namespaced â€”
`Delete` is registered 4Ă— (from C++ `AttractionPtDelete`, `DeleteFilter`, `FocusPtDelete`,
`DeleteSquad`), `Enable` 4Ă—, `Create` 3Ă—. A flat registry would clobber them.

**Confidence: confirmed** for "no static table" (exhaustive scan), the stanza shape, the
per-table registry, and the 26/898 grouping (all byte-level and mechanically reproduced).

## The name lives in two places, and they differ

Each binding's Lua name is a plain `.rdata` C string, stored **twice** per stanza: once into
`node+4` and once into a per-instantiation `static const char* s_name` slot in `.data` (898
distinct slots spanning 0x01241f88-0x0142e628). The C++ symbol lives separately, in the RTTI
mangled name.

They are not the same string. **256 of 898 differ**, and the difference is exactly the
namespace prefix the C++ side carries and the Lua side does not:

| C++ symbol (in `lua_bindings.txt`) | actually callable as |
|---|---|
| `ActorRagdoll` | `Actor.Ragdoll` |
| `ActorIsAlive` | `Actor.IsAlive` |
| `ActorGetPosition` | `Actor.GetPosition` |
| `PlayerKillModeCancel` | `Actor.ExitSpecialKillMode` |
| `TrainRegisterTrainDecoupledCallback` | `Train.RegisterDecoupledCallback` |
| `SetDistantRagdollSound` | `Actor.SetDistantRagdollSound` (unchanged) |

The last row matters: the prefix is *not* mechanically strippable. `SetDistantRagdollSound` and
`ActorRagdoll` are both in `Actor.cpp` and both land in the `Actor` table, but only one carried
a prefix in C++. The mapping must be read from the binary, which is what the tool does.

Independent confirmation from the Lua corpus: `Actor.Ragdoll(hVitt)` at
[`Missions/Act_1_Farm.lua:755`](../saboteur-luacd/src/Missions/Act_1_Farm.lua) and
`Actor.SetDistantRagdollSound(hDropNazi1)` at
[`Missions/Act_3_Mission_2.lua:1031`](../saboteur-luacd/src/Missions/Act_3_Mission_2.lua).

**Confidence: confirmed** for the mapping itself. **Inferred** for *why* the prefix exists â€”
most likely the C++ names predate the table split and were never renamed; I have no direct
evidence.

### Corpus cross-check

Harvesting every `Table.Func(` call site from the 321-file Lua corpus gives 708 calls whose
table is one of the 26. **649 (91.7%) resolve against the map.** The 59 that don't are
explained, not silent failures: several are Lua-side module tables that shadow an engine table
name â€” `Experimental/Checkpoint.lua` defines `Checkpoint:EnterState_*` methods on a *Lua* table
that happens to share the engine's `Checkpoint` name. Others (`Suspicion.SetState`,
`Util.FindObjectHandle`) have no definition anywhere in the corpus and are most likely
dev-build bindings cut from retail â€” the corpus is decompiled *development* script, the binary
is retail. I have not proven that case-by-case.

**Confidence: confirmed** for the 91.7% figure; **inferred** for the cut-binding explanation.

## How the surface is enumerated at boot

The 26 initializers are **registered, not called**, from the CRT-init region. For the `Actor`
initializer, the only reference to 0x0070caf0 in the whole image is at 0x00f56adf:

```
68 f0 ca 70 00     push 0x0070caf0          ; the Actor initializer
b9 84 d7 42 01     mov  ecx, 0x0142d784     ; a global
e8 62 35 7a ff     call 0x006fa050
c3                 ret
```

`FUN_006fa050` stores the pointer into the global and tail-calls `FUN_004ec67a`. So the
initializers are queued on a deferred-init object at CRT time and drained later, rather than
running the registration at static-init time.

I could **not** pin the drain point, and one specific thing blocks it: `FUN_006f6620` â€” the
registry constructor that receives the table name â€” is `jmp 0x01638da0`, and 0x01638da0 lies
inside the **`.secu` section** (0x015fc000-0x0163ca40). This is a SecuROM-protected binary and
the registry ctor body has been relocated behind the protection. The build-time side of the
seam is fully readable; the consumption side (registry list â†’ real Lua table) is not, statically.

**Confidence: confirmed** for the CRT-init stub bytes and the `.secu` jmp target.
**Open** for the drain order and for what the 0x3c-byte registry actually does with the list.

## Relation to `FUN_006f96e0` / `DAT_0142d324`

Both are **outside** this mechanism, and I want to be explicit rather than force a connection:

- `FUN_006f96e0` is 7 bytes â€” `{ FUN_004d2941(); return; }`, a pure forwarder. It appears in the
  *callee* side of many bindings, not in registration. It belongs to
  [02-marshalling-abi.md](02-marshalling-abi.md).
- `DAT_0142d324` is **not** one of the 898 `s_name` slots (checked against the full set). It sits
  in the same `.data` neighbourhood as them (0x01241f88-0x0142e628) but is a distinct global.
  Adjacency here is a linker artifact and should not be read as a relationship.

The registration path touches two different globals worth naming: `0x0142d784` (the deferred-init
object the `Actor` initializer is queued on) and the per-binding `s_name` slots.

**Confidence: confirmed** that `DAT_0142d324` is not an `s_name` slot. **Open** as to what it is.

## Recipe: dump the full {name â†’ VA} table

```
python tools/dump_lua_registration.py "C:/GOG Games/The Saboteur/Saboteur.exe" \
    -o data/lua_registration_map.tsv
```

Needs only Python 3 and the retail `Saboteur.exe` â€” no Ghidra project, no capstone, ~20s. It
prints a summary (899 instantiations / 898 stanzas / 26 runs / 26 tables) and writes 898 rows:

```
table  lua_name  cpp_symbol  cpp_class  family  shape  nresults  impl_va  thunk_va  stub_va  vtable_va  s_name_slot  stanza_va
Actor  Ragdoll   ActorRagdoll          LuaGlueFunctor0   adapter  1    00714230  00716b20  00717390  00fde930  0142d81c  0070d71d
Util   GetHandleByName  GetHandleByName  LuaGlueFunctor0R  jmp   eax  00758b30  0075d460  0075d580  00fe7a2c  0142e254  00754535
```

Method, in order: RTTI type descriptors containing `LuaGlue` â†’ COLs whose `+0x0c` points at one
â†’ vtable at `COL+4` â†’ getter stub `mov eax,<thunk>; ret` â†’ thunk (3 shapes) â†’ impl; separately,
`c7 00 <vtable> / c7 05 <slot> <nameptr>` gives the Lua name, and back-scanning each run for
`push <str>` + `call 0x006f6620` gives the table.

Two hardcoded VAs make it build-specific: `0x006f6620` (registry ctor) and `0x006f6660` (list
insert). It will not port to another build or to the 2008 prototype without re-deriving those.

## Open questions

1. **What is `mov eax, 1` in the adapter?** Every `void`-binding adapter returns the constant 1.
   If it is `lua_CFunction`'s `nresults`, a `void` binding claiming one result is odd; it may
   instead be a `bool` "handled" convention with the real dispatch one level up. I did not
   confirm the calling convention against the Lua VM and am not going to guess it. This is the
   most load-bearing open item; it belongs to [02-marshalling-abi.md](02-marshalling-abi.md).
2. **When are the 26 registries drained into real Lua tables, and in what order?** Blocked
   statically by the `.secu` relocation of `FUN_006f6620`. A runtime breakpoint (the x32dbg MCP
   tooling is available) on 0x006f6660 would settle it and also confirm the registry layout.
3. **What is the 0x3c-byte registry?** Only `+0x08` (list head) and `+0x10` (count) are known,
   from `FUN_006f6660`. The other 0x30 bytes are unread.
4. **Are the 59 unresolved corpus calls really cut bindings?** Testable against the 2008
   pre-release build (tracked as a separate lead, not yet downloaded) â€” if `Suspicion.SetState`
   registers there, that settles it.
5. **`DAT_0142d324`** â€” not an `s_name` slot, otherwise unidentified here.
6. **Why do 256 C++ symbols carry a table prefix and 642 not?** No evidence either way.
